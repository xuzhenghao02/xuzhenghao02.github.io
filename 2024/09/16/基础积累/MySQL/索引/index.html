索引的组织结构

一个节点能存储的索引数量越多，树的高度越低，查询复杂度越小

索引的存储结构

  • 聚簇索引
    • 提升IO密集型应用性能
    • Row的移动不影响二级索引
    • 如果主键值非自增，在插入记录时，可能造成记录移动或页分裂，影响性能
  • 非聚簇索引
    • 索引与数据分离，相对聚簇索引占用空间小
    • InnoDB二级索引相对MyISAM会大
    • 使用InnoDB二级索引需要检索两次B+树
    • Row的移动会影响MyISAM索引
    • 随机IO

例子：MySQL索引的理解（主键索引和二级索引）

简单来说，InnoDB中，表的主键索引是聚簇索引，索引和数值存储在同一节点，其余索引是非聚簇索引，只存储与主键索引的映射关系；MyISAM主键和其他索引没有区别看，都是非聚簇索引

索引下推

  • 个人理解：可以理解为有一个多级索引（a，b）等，其中a查询条件是范围查询，同时会涉及b的精确查询
    • 如果不使用索引下推，会扫描B+树，把符合a取值范围的id全部找出来进行回表，回表之后拿到所有数据，再对b条件进行过滤
    • 如果使用索引下推，当扫描B+树的时候，利用联合索引的数据，对a和b同时进行筛选过滤，将同时符合双方条件的id进行回表，这种情况下回表的数据将大大减少（由于增加了b的过滤条件）
  • 文章：MySQL查询性能优化七种武器之索引下推
  • 原理：本来在MySQL Server层做的筛选操作，下推到存储引擎层来做
  • 优点
    1. 减少回表查询的次数
    2. 减少存储引擎和MySQL Server层的数据传输量
  • 使用条件
    • 只能用于range、 ref、 eq_ref、ref_or_null访问方法
    • 只能用于InnoDB和 MyISAM存储引擎及其分区表
    • 对InnoDB存储引擎来说，索引下推只适用于二级索引（也叫辅助索引）
    • 引用了子查询的条件不能下推
    • 引用了存储函数的条件不能下推，因为存储引擎无法调用存储函数
  • 例子
    select name,age from table where name like ‘张%’ and age =18

设计索引

  • WHERE、JOIN、GROUP BY、ORDER BY等语句使用的列建议使用索引
  • 使用最左前缀：（A,B,C）只能匹配A、A&B、A&B&C
  • 如果选择索引列的顺序
    1. 经常使用的列优先：使用最左前缀，尽量复用已有索引
    2. 区分度大的列优先：区分度 = distinct(col) / count(col)，distinct即cardinality，至少小于0.1
    3. 宽度小的列优先：列宽度 = 列的数据类型，宽度越小，单节点的key越多，索引树的高度越高，查询复杂度越低
